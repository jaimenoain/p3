## CURRENT_ARCHITECTURE_SNAPSHOT

The active Walking Skeleton uses the Next.js App Router with three route groups. **Public:** `/` (landing teaser). **Auth** `(auth)`: `/login`, `/signup`, `/forgot-password` with Server Actions and Supabase; centered full-screen layout. **Dashboard** `(dashboard)`: `/dashboard`, `/assets`, `/airlock`, `/vault`, `/governance`, `/settings`; single layout with Sidebar (from `@/components/ui/sidebar`), top header, and scrollable main content. **Guest** `(guest)`: `/guest/view` for time-boxed access. Middleware (`src/middleware.ts`) protects dashboard and guest routes and redirects unauthenticated users to `/login`; authenticated users hitting `/` or `/login` are redirected to `/dashboard`. Root layout wraps the app in `AuthProvider` (React Context for `session` and `isLoading` via `useAuth()`). No `/api/v1/*` or workspace/scenario routes yet; those are Phase 2+.

## Phase 1.2 — Auth loop (complete)

### Supabase/Resend Auth loop & global session state

- **Log In (`/login`):** Server Action `loginAction` calls `supabase.auth.signInWithPassword`; on success redirects to `/dashboard`, on failure returns generic "Invalid email or password". UI: Shadcn `Button`, `Input`, `useActionState` for errors.
- **Sign Up (`/signup`):** Server Action `signUpAction` calls `supabase.auth.signUp`; redirects to `/login` after signup. Data contract: User table row must be created via DB trigger or post-signup hook.
- **Forgot Password (`/forgot-password`):** Server Action `forgotPasswordAction` calls `supabase.auth.resetPasswordForEmail`; minimalist flow with success message.
- **Log Out:** Sidebar logout button submits form with Server Action `logoutAction`; calls `supabase.auth.signOut()` and redirects to `/login`.
- **Session state:** `AuthProvider` (root layout) consumes Supabase browser client, fetches session on mount, subscribes to `onAuthStateChange`, exposes `session` and `isLoading` via React Context (`useAuth()`). Middleware enforces default-deny; layouts are conditionally effective via middleware redirects (unauthenticated → `(auth)`; authenticated on `/` or `/login` → `/dashboard`).
- **Verification:** `node test/verify_task_1_2.mjs` (all 8 checks pass).

### Multi-tenant data architecture & auto-provisioning

- **Schemas in use:** PostgreSQL tables `users`, `organizations`, `organization_members`, `workspaces`, and `scenarios` (with RLS) are implemented in the live database. The app assumes `public.users` is populated by a Postgres trigger on signup.
- **Auto-provisioning:** On first load of the `(dashboard)` layout, if the authenticated user has no `organization_members` row, the app runs `provisionTenantAction` (Server Action in `app/(dashboard)/actions.ts`). It creates in order: one default organization ("My Startup"), one `organization_members` row (current user as owner), one workspace ("Primary Workspace"), and one baseline scenario (`is_active_baseline: true`, `global_assumptions: {}`). `organization_members.user_id` is set to the current auth user so RLS is satisfied.
- **UX:** Users without a workspace see a full-screen loading state ("Setting up your workspace..." with Loader2 spinner) until provisioning completes, then `router.refresh()` and the standard dashboard shell (Sidebar + children) render.
