## CURRENT_ARCHITECTURE_SNAPSHOT

The app uses the Next.js App Router with three route groups. **Public:** `/` (landing teaser). **Auth** `(auth)`: `/login`, `/signup`, `/forgot-password` with Server Actions and Supabase; centered full-screen layout. **Dashboard** `(dashboard)`: `/dashboard`, `/assets`, `/airlock`, `/vault`, `/governance`, `/settings`; single layout with Sidebar (from `@/components/ui/sidebar`), top header, and scrollable main content. **Guest** `(guest)`: `/guest/view` for time-boxed access. Middleware (`middleware.ts`) uses the Supabase server client and protects dashboard and reserved application prefixes (`/dashboard`, `/canvas`, `/import`, `/actuals`, `/settings`, `/assets`, `/airlock`, `/vault`, `/governance`) plus guest routes (`/guest`), redirecting unauthenticated users to `/login` and redirecting authenticated users from `/` or any auth path back to `/dashboard`. Root layout wraps the app in `AuthProvider` (React Context for `session` and `isLoading` via `useAuth()`).

Within `(dashboard)`, the layout is now tenant-aware. On each request it uses the Supabase server client to fetch the current user, their first `organization_members` row, the owning `organizations` record, and the first `workspaces` row for that organization, all subject to Postgres RLS. If no membership exists, it renders `DashboardProvisioningGate`, a client component that calls `provisionTenantAction`, which in turn invokes the SECURITY DEFINER `public.provision_tenant(new_user_id, org_name, workspace_name)` RPC. The RPC creates a single-tenant slice (organization, membership, workspace with `starting_cash_balance` set to NULL, and a baseline scenario) bound to `auth.uid()`. Once a workspace exists, the layout constructs a simple workspace context used by the Sidebar and header; if `starting_cash_balance` is NULL, it overlays `CashBalancePrompt`, a blocking dialog wired to `updateWorkspaceCashBalanceAction` that validates a non-negative number and updates `workspaces.starting_cash_balance` under workspace RLS, then revalidates `/dashboard`. There are still no `/api/v1/*` HTTP routes or Projection Canvas / Output Dashboard UIs; those remain future phases.

### Multi-tenant data architecture & auto-provisioning

- **Schemas in use:** PostgreSQL tables `users`, `organizations`, `organization_members`, `workspaces`, and `scenarios` (with RLS) are implemented in the live database. The app assumes `public.users` is populated by a Postgres trigger on signup.
- **Auto-provisioning:** On first load of the `(dashboard)` layout, if the authenticated user has no `organization_members` row, the app runs `provisionTenantAction` (Server Action in `app/(dashboard)/actions.ts`). - **UX:** Users without a workspace see a full-screen loading state ("Setting up your workspace..." with Loader2 spinner) until provisioning completes, then `router.refresh()` and the standard dashboard shell (Sidebar + children) render.

### Blocks & Projection Canvas — Dependency Logic & Recalculation COMPLETE

- **Reference fields & payload structure:** Extended the `blocks.payload` JSONB schema with a `dependencies` map whose values follow the input-mode contract `{ mode: 'Static' | 'Referenced' | 'Formula', value?: number, referenceId?: string, formula?: string }`. Server-side Zod schemas (`NumericInputConfigSchema`) and DTOs now describe this structure so the Projection Engine can distinguish static inputs from referenced or formula-driven fields.
- **Server-side dependency updates:** Added `updateBlockDependencyMutation({ blockId, field, mode, referenceId?, formula?, value? })` in `app/(dashboard)/actions.ts`. It runs under the authenticated Supabase session, respects existing `blocks` RLS, builds a block-level dependency graph from `payload.dependencies`, and performs cycle detection before persisting changes. Attempts that would introduce a cycle are rejected with the exact message "Circular dependency detected. This connection is not allowed."
- **Canvas dependency UI & 30/360 recalculation lock:** Updated the `/canvas` `ProjectionCanvasClient` and `BlockCard` to include a "Starting MRR input mode" selector and a dependency dropdown for Revenue blocks. Eligible parent blocks appear in the dropdown; blocks that are already downstream in the dependency graph (or the block itself) are shown but disabled, preventing circular selection. All dependency and property mutations are wrapped in a canvas-level recalculation overlay that locks the canvas for at least ~1 second with the text "Recalculating model...", matching the 30/360-based engine’s synchronous recompute UX.

### Blocks & Projection Canvas — Soft Disable & Inactives Tray COMPLETE

- **Soft disable backend:** Implemented `updateScenarioBlocksMutation({ blockId, isActive })` in `app/(dashboard)/actions.ts`. It runs under the authenticated Supabase session, relies on `blocks` RLS to ensure workspace/organization membership, and updates `blocks.is_active` for the specified block. The projection engine will treat `is_active = false` blocks as having zero financial impact when computing scenario outputs.
- **On/Off toggle UI:** Added a Shadcn-style `Switch` to the header of each block card in the `/canvas` Projection Canvas.
